Phase 2
Study Guide
JWT authentication, bcrypt passwords, SQLAlchemy, React Router, and role-based access — explained line by line.
What Phase 2 adds on top of Phase 1
Phase 1 was a stateless chatbot — anyone could use it. Phase 2 adds identity: users register, log in, receive a signed token, and that token gates every API call. The streaming chat logic is completely unchanged — we just wrap a security layer around it.

The three big new concepts

JWT
A signed string the server gives after login. The client sends it with every request. No database lookup needed to verify — just check the signature.
bcrypt
A one-way password hash. You can never recover the original — only verify a guess. Intentionally slow to defeat brute-force attacks.
SQLAlchemy ORM
Maps Python classes to database tables. Write Python, not SQL. Handles inserts, queries, and updates automatically.
Auth Flow Pipeline
Every step from submitting the login form to an authenticated AI chat request.

Registration flow

1
User fills in email + password in Login.jsx
React local state tracks the form fields. On submit, e.preventDefault() stops the browser reload, then handleRegister(email, password) from useAuth is called.
2
useAuth calls register() in api/auth.js
axios.post('/api/auth/register', { email, password }) is sent. Axios uses the shared apiClient instance which has the base URL and interceptors already configured.
POST /api/auth/register → { email, password, full_name }
3
FastAPI validates the request with Pydantic
RegisterRequest schema checks: is email valid? Is password at least 8 chars? The @field_validator runs automatically. If validation fails, a 422 error is returned before our code even runs.
4
auth_service.register_user() checks for duplicates
Queries the database for an existing user with this email. If found, raises HTTPException(400). If not, hashes the password with bcrypt and creates a new User record.
db.query(User).filter(User.email == email).first()
5
User is saved to SQLite, response returned
db.add(user) → db.commit() → db.refresh(user). The UserResponse schema strips out hashed_password before serialising — only safe fields go back to the client.
6
useAuth auto-logs in after registration
After register succeeds, handleRegister immediately calls login(email, password) so the user doesn't have to submit the form twice.

Login flow

1
POST /api/auth/login with email + password
FastAPI validates the LoginRequest schema. The route handler calls login_user(request, db) from auth_service.
2
auth_service fetches user and verifies password
db.query(User).filter(User.email == email).first() finds the user. verify_password(plain, hashed) runs bcrypt comparison. Both wrong-email and wrong-password return the same 401 error — preventing email enumeration attacks.
verify_password("mypass", "$2b$12$...") → True/False
3
JWT token is created and returned
create_access_token({ "sub": str(user.id), "role": user.role }) signs the payload with the secret key. The token string is returned along with the role (so React doesn't need to decode the JWT).
→ { access_token: "eyJ...", role: "admin", user_id: 1 }
4
React stores token in Zustand + localStorage
setAuth(token, user) in authStore saves to both. localStorage means the token survives a browser refresh — without it, the user would be logged out every time they reload the page.
localStorage.setItem('auth_token', token)
5
React Router navigates to /chat
navigate(from, { replace: true }) redirects to the original destination. If the user went directly to /chat before logging in, they land back there. Otherwise they go to /chat by default.

Authenticated request flow (chat)

1
User sends a chat message
Same as Phase 1 — sendMessage() in useChat is called. The only difference: api/chat.js now reads the token from authStore and adds it to the fetch header.
2
fetch() includes Authorization header
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9... is added to every streaming fetch request. FastAPI reads this header via the HTTPBearer scheme.
headers: { 'Authorization': `Bearer ${token}` }
3
FastAPI's get_current_user() dependency runs
The dependency chain: HTTPBearer extracts the token string → decode_access_token() verifies the JWT signature and expiry → db.query(User).filter(User.id == id).first() fetches the user record. If any step fails, 401 is raised before the route handler runs.
4
Route handler receives the User object
The User's role determines which system prompt the AI gets. Admin sees full capability, customer sees restricted support bot — no extra API calls needed, it's all in the token.
system_prompt = SYSTEM_PROMPTS[current_user.role]
security.py ★
The two pillars of auth security: bcrypt password hashing and JWT token management.
Python JWT bcrypt

Password hashing with bcrypt

⚠️ Why you must never store plain passwords
Databases get leaked. If passwords are stored in plain text, every user's password is exposed. With bcrypt, even if the database is stolen, attackers can't recover passwords — they'd need to guess each one, which takes years per password given bcrypt's intentional slowness.
backend/app/core/security.py — password section
from passlib.context import CryptContext pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")
Create one shared CryptContext. This object manages hashing. schemes=["bcrypt"] means use bcrypt algorithm. deprecated="auto" means if you ever switch algorithms in the future, old hashes are automatically re-hashed on next login. Always create this once at module level — not inside each function.
def hash_password(plain: str) -> str: return pwd_context.hash(plain)
One-way hash a plain password. pwd_context.hash("mypassword") returns something like $2b$12$EixZaYVK.... This is what gets stored in the database. The result is different every call (bcrypt adds a random "salt") — two identical passwords produce different hashes. Why bcrypt and not SHA-256? SHA-256 is fast — designed for data integrity. bcrypt is designed to be slow for exactly this security reason.
def verify_password(plain: str, hashed: str) -> bool: return pwd_context.verify(plain, hashed)
Check if a plain password matches a stored hash. passlib extracts the salt from the stored hash, re-hashes the plain password with that salt, and compares. Returns True if they match, False otherwise. Used at login time — you never "decrypt" a bcrypt hash, you only verify a guess against it.

JWT tokens

What is a JWT?
A JWT (JSON Web Token) is a base64-encoded string with three parts separated by dots: Header.Payload.Signature. The header says which algorithm was used. The payload contains the data (user ID, role, expiry). The signature is a hash of the header + payload signed with your secret key. Anyone can read the payload — but only someone with the secret key can create a valid signature.
backend/app/core/security.py — JWT section
def create_access_token(data: dict, expires_delta=None) -> str: to_encode = data.copy() expire = datetime.now(timezone.utc) + ( expires_delta or timedelta(minutes=settings.access_token_expire_minutes) ) to_encode.update({"exp": expire}) return jwt.encode(to_encode, settings.jwt_secret_key, algorithm="HS256")
Create a signed JWT token. We copy the data dict (never mutate the original), add an exp (expiry) timestamp, then call jwt.encode(). The result is a string like eyJhbGci.... data typically contains {"sub": "42", "role": "admin"} — sub is a JWT standard field for the subject (user ID). HS256 means HMAC-SHA256 — a symmetric algorithm. The same secret key creates AND verifies tokens. For public APIs, RS256 (asymmetric) is more secure.
def decode_access_token(token: str): try: payload = jwt.decode(token, settings.jwt_secret_key, algorithms=["HS256"]) return payload except JWTError: return None
Verify and decode a JWT. jwt.decode() checks: (1) the signature is valid — token wasn't tampered with, (2) the token hasn't expired. If either check fails, JWTError is raised — we catch it and return None. Callers check for None and raise a 401. No database call needed — the signature verification is pure cryptography.
database.py
SQLAlchemy setup — the engine, session factory, Base class, and get_db dependency.
Python SQLAlchemy

The three components

backend/app/core/database.py
engine = create_engine( settings.database_url, connect_args={"check_same_thread": False} )
The engine manages database connections. Think of it as the phone line to the database. database_url is sqlite:///./chatbot.db — SQLite creates a file on disk. check_same_thread: False is SQLite-specific — FastAPI's async nature uses multiple threads and this allows it. For PostgreSQL, remove connect_args entirely and change the URL to postgresql://user:pass@localhost/dbname.
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
SessionLocal is a factory that creates DB sessions. A session is a "unit of work" — it tracks all changes and commits or rolls them back together. autocommit=False means we manually call db.commit(). autoflush=False means SQLAlchemy doesn't automatically send SQL to the DB until we call commit. This gives us full control.
Base = declarative_base()
Base is the parent class all models inherit from. SQLAlchemy uses it to track which Python classes map to which database tables. When we call Base.metadata.create_all(engine) in main.py, it scans all Base subclasses and creates their tables if they don't exist yet.
def get_db(): db = SessionLocal() try: yield db finally: db.close()
FastAPI dependency that provides a session per request. This is a generator function (uses yield) which lets FastAPI manage the lifecycle. The finally block guarantees db.close() runs even if an exception is thrown mid-request. Without closing, database connections would leak over time. Route handlers use it as db: Session = Depends(get_db).
models/user.py
The User SQLAlchemy model — maps a Python class to the users database table.
Python SQLAlchemy
backend/app/models/user.py
class User(Base): __tablename__ = "users"
Inheriting from Base registers this as a SQLAlchemy model. __tablename__ sets the actual database table name. If you change this string, the table is renamed (after you drop and recreate it). This class is both a Python object AND a blueprint for a database table.
id = Column(Integer, primary_key=True, index=True) email = Column(String, unique=True, index=True, nullable=False)
Defining columns. primary_key=True makes this the unique row identifier — auto-increments. index=True creates a database index for fast lookups. unique=True prevents two users having the same email (enforced at the DB level, not just our code). nullable=False means the column is required — trying to save a User without an email throws a DB error.
hashed_password = Column(String, nullable=False) role = Column(String, default="user", nullable=False)
Password hash and role. We store the bcrypt hash, never the plain password. role has default="user" — every new account starts as a regular user. The "least privilege" principle: grant the minimum access by default, elevate manually. Valid roles: "admin", "user", "customer". In Phase 3 this drives the RoleGuard and system prompt selection.
created_at = Column(DateTime(timezone=True), server_default=func.now())
server_default vs default. server_default=func.now() tells the database to set this timestamp using the database's own clock when inserting a row. This is more reliable than Python's datetime.now() — it doesn't depend on the server's timezone and is consistent even if multiple Python processes are running.
dependencies.py ★
FastAPI's Depends() system — reusable auth guards that run before route handlers.
Python FastAPI
What is a FastAPI Dependency?
A dependency is a function you declare with Depends() in a route parameter. FastAPI calls it automatically before your route handler runs. It's like middleware but per-route — only runs for routes that need it. Dependencies can depend on other dependencies, forming a chain.
backend/app/core/dependencies.py
bearer_scheme = HTTPBearer(auto_error=False)
HTTPBearer reads the Authorization header automatically. It expects the format Authorization: Bearer <token>. auto_error=False means it returns None instead of raising a 403 when the header is missing — we raise our own 401 with a better error message. The 403 vs 401 distinction matters: 401 = not authenticated, 403 = authenticated but not authorised.
def get_current_user( credentials = Depends(bearer_scheme), db: Session = Depends(get_db), ) -> User:
A dependency that depends on other dependencies. FastAPI resolves the whole chain: it runs bearer_scheme to extract the token, runs get_db to get a DB session, then passes both to get_current_user. You never call these functions yourself — FastAPI's dependency injection system handles it all.
if not credentials: raise HTTPException(401, "Not authenticated.") payload = decode_access_token(credentials.credentials) if payload is None: raise HTTPException(401, "Token invalid or expired.")
Three-step validation. (1) Was an Authorization header present? (2) Is the JWT signature valid and not expired? (3) Does the user ID in the token match a real user in the DB? All three must pass. Raising HTTPException stops execution immediately — the route handler never runs.
def require_role(required_role: str): def checker(current_user = Depends(get_current_user)) -> User: if current_user.role != required_role: raise HTTPException(403, "Access denied.") return current_user return checker
Factory function pattern. require_role("admin") returns a NEW function (checker) that FastAPI can use as a dependency. checker itself depends on get_current_user — so auth is checked first, then role. This is a closure: checker "closes over" the required_role variable. Usage: user = Depends(require_role("admin")).
auth_service.py ★
Business logic for registration and login — the layer between HTTP routes and the database.
Python SQLAlchemy
Why separate service from routes?
The route (api/auth.py) handles HTTP — reading the request body, returning the right response code. The service (auth_service.py) handles business rules — how to register, how to validate, what constitutes a valid login. This separation means you could add a CLI command that also registers users by just calling the service function — no HTTP needed.
backend/app/services/auth_service.py — login_user()
INVALID_CREDENTIALS_ERROR = HTTPException( status_code=401, detail="Incorrect email or password.", )
Same error for wrong email AND wrong password. This is intentional security design. If wrong-email returned "email not found" and wrong-password returned "wrong password", an attacker could use different error messages to discover which emails are registered. By returning identical errors, we prevent email enumeration.
user = db.query(User).filter(User.email == request.email).first() if not user: raise INVALID_CREDENTIALS_ERROR if not verify_password(request.password, user.hashed_password): raise INVALID_CREDENTIALS_ERROR
SQLAlchemy query chain. db.query(User) starts a query on the users table. .filter(User.email == email) adds a WHERE clause. .first() returns the first result or None. This translates to SQL: SELECT * FROM users WHERE email = ? LIMIT 1. SQLAlchemy writes the SQL for you — you write Python.
token = create_access_token(data={ "sub": str(user.id), "role": user.role, "email": user.email, })
What goes in the JWT payload. sub (subject) is a JWT standard field — we put the user ID here as a string (JWT requires strings). We also include role and email as convenience — the frontend can read these without making another API call. The payload is base64-encoded, not encrypted — don't put sensitive data here.
api/auth.py
The three auth HTTP endpoints: register, login, and /me.
Python FastAPI
backend/app/api/auth.py
@router.post("/register", response_model=UserResponse, status_code=201) async def register(request: RegisterRequest, db: Session = Depends(get_db)): new_user = register_user(request, db) return new_user
Thin route — all logic in the service layer. status_code=201 sets the HTTP status to 201 Created (vs. 200 OK) — semantically correct for resource creation. response_model=UserResponse tells FastAPI to serialise the return value using that Pydantic schema, automatically stripping hashed_password and any other field not in UserResponse.
@router.get("/me", response_model=UserResponse) async def get_me(current_user: User = Depends(get_current_user)): return current_user
The /me endpoint pattern — common in auth systems. The route handler has literally zero auth logic — Depends(get_current_user) handles everything. If the token is valid, the User object is handed to the function. If not, the function never runs. This endpoint lets the frontend verify the token is still valid and get the latest user data on app load.
authStore.js ★
Global auth state with localStorage persistence. The frontend's source of truth for "who is logged in."
React Zustand
⚠️ Why localStorage for the token?
Without localStorage, every browser refresh logs the user out — the token only lives in memory. With localStorage, the token survives refreshes. The downside: localStorage is accessible to JavaScript on the page, making it vulnerable to XSS attacks. For highly sensitive apps, httpOnly cookies are more secure — but for a learning project, localStorage is the standard approach.
frontend/src/store/authStore.js
const loadStoredAuth = () => { try { const token = localStorage.getItem('auth_token') const user = JSON.parse(localStorage.getItem('auth_user') || 'null') return { token, user } } catch { return { token: null, user: null } } }
Load persisted auth on app startup. This runs once when the module is first imported. We wrap in try/catch because JSON.parse throws if the stored value is malformed (e.g. if localStorage was partially corrupted). The fallback returns null — the user will need to log in again.
setAuth: (token, user) => { localStorage.setItem('auth_token', token) localStorage.setItem('auth_user', JSON.stringify(user)) set({ token, user, error: null, isLoading: false }) },
Always save to both localStorage AND Zustand. localStorage is the persistence layer (survives refresh). Zustand is the reactive layer (triggers React re-renders). You need both: if you only save to localStorage, components don't re-render on login. If you only save to Zustand, a refresh loses the token.
logout: () => { localStorage.removeItem('auth_token') localStorage.removeItem('auth_user') set({ token: null, user: null }) },
Remove from both layers on logout. Calling just set({ token: null }) would clear Zustand but leave the token in localStorage — the next refresh would log the user back in. Always clear both.
api/auth.js ★
Axios instance with interceptors that automatically attach tokens and handle 401 responses globally.
JavaScript Axios
Interceptors vs manual token attachment
Without interceptors, every API call file would need: headers: { Authorization: `Bearer ${token}` }. With the request interceptor, you write it once and every Axios call in the whole app gets the token automatically. The response interceptor catches 401s globally — one place to handle "your session expired."
frontend/src/api/auth.js
export const apiClient = axios.create({ baseURL: '/api', headers: { 'Content-Type': 'application/json' }, })
One shared Axios instance for the entire app. axios.create() creates a new Axios instance with preset config. All calls using apiClient share the baseURL and headers — you write apiClient.get('/auth/me') instead of the full URL every time. The streaming fetch in api/chat.js doesn't use this because it needs native fetch for ReadableStream support.
apiClient.interceptors.request.use((config) => { const token = useAuthStore.getState().token if (token) config.headers.Authorization = `Bearer ${token}` return config })
Request interceptor: attach JWT to every request. This runs before every Axios call. useAuthStore.getState() reads Zustand directly (not via hook) — interceptors are outside React components so hooks can't be used here. getState() is Zustand's escape hatch for non-component code. We modify config.headers and must return the config — returning nothing breaks the request.
apiClient.interceptors.response.use( (response) => response, (error) => { if (error.response?.status === 401) { useAuthStore.getState().logout() window.location.href = '/login' } return Promise.reject(error) } )
Response interceptor: global 401 handler. The first function handles success (pass through). The second handles errors. A 401 means the token expired or was invalidated — we log out and force redirect to login. window.location.href is a hard redirect (reloads the page) rather than React Router's navigate() — this ensures all state is reset. We still call Promise.reject(error) so the original caller can also handle the error if needed.
ProtectedRoute.jsx ★
Redirects unauthenticated users to /login. The React-side gate for private pages.
React React Router
Security reminder
ProtectedRoute is UI-only security. A determined user could remove it from the DOM with browser DevTools. The real security is in the backend — FastAPI's get_current_user dependency rejects any API call without a valid JWT. ProtectedRoute is for user experience, not security enforcement.
frontend/src/guards/ProtectedRoute.jsx
export function ProtectedRoute({ children }) { const token = useAuthStore((state) => state.token) const location = useLocation()
Read token from Zustand, current URL from React Router. We read token (not the whole state) so this component only re-renders when the token changes — not on every store update. useLocation() gives us the current URL path so we can redirect back after login.
if (!token) { return <Navigate to="/login" state={{ from: location }} replace /> }
Navigate component for declarative redirect. state={{ from: location }} passes the current path to the login page — after login, useAuth reads location.state?.from?.pathname and redirects back to the original page. replace replaces the browser history entry — so pressing Back after login doesn't go back to the protected page (which would just redirect again).
return children }
If authenticated, render the children. The ProtectedRoute component is transparent when logged in — it just passes through whatever is wrapped inside it. In App.jsx: <ProtectedRoute><Chat /></ProtectedRoute> — if no token, redirect. If token, render Chat.

Usage in App.jsx

RoleGuard.jsx
Conditionally renders children based on the current user's role. Used for showing/hiding UI elements.
React
frontend/src/guards/RoleGuard.jsx
export function RoleGuard({ role, roles, children, fallback = null }) { const user = useAuthStore((state) => state.user) if (!user) return fallback
Accepts either a single role or an array of roles. role="admin" for one role, roles={["admin","user"]} for multiple. fallback is what renders when access is denied — defaults to null (renders nothing). You could pass a "Upgrade to Pro" message as the fallback.
const allowedRoles = roles || (role ? [role] : []) const isAllowed = allowedRoles.includes(user.role) if (!isAllowed) return fallback return children }
Normalise to an array, then check inclusion. Whether you pass role="admin" or roles={["admin"]}, the check is the same: is user.role in the allowed list? Returns children if yes, fallback if no. Usage: <RoleGuard role="admin"><AdminPanel /></RoleGuard>.
useAuth.js
Custom hook connecting authStore + API calls into one clean interface for components.
React
frontend/src/hooks/useAuth.js
const from = location.state?.from?.pathname || '/chat'
The "redirect back" pattern. When ProtectedRoute redirects to /login, it passes the original URL in location.state.from. Here we read it: if state exists, go back there after login; otherwise default to /chat. The ?. optional chaining prevents errors if state or from is undefined.
const handleLogin = useCallback(async (email, password) => { setLoading(true) clearError() try { const data = await login(email, password) setAuth(data.access_token, { id: data.user_id, role: data.role, email }) navigate(from, { replace: true }) } catch (err) { setError(err.response?.data?.detail || 'Login failed.') } }, [...])
The full login flow in one function. Set loading → clear previous error → call API → on success: save auth, navigate. On failure: show the error from FastAPI's response body (err.response?.data?.detail). The optional chaining handles network errors where err.response may be undefined. useCallback memoises the function so it doesn't recreate on every render.

Login.jsx key patterns

frontend/src/pages/Login.jsx
const [mode, setMode] = useState('login') const [email, setEmail] = useState('') const [password, setPassword] = useState('')
Local state for the form — not in Zustand. Form field values only matter on this page. There's no reason to share them globally. Zustand is for state that multiple components need. useState is for state that belongs to one component. This is the right choice here.
const handleSubmit = async (e) => { e.preventDefault() if (mode === 'login') await handleLogin(email, password) else await handleRegister(email, password, fullName) }
e.preventDefault() is critical on form submit. Without it, the browser does its default form submission — navigates to the URL in the form's action attribute, causing a full page reload. This clears all React state and sends data as a query string. In React, you always prevent default and handle submission yourself with JavaScript.
