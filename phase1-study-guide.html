What are we building?

A real-time AI chatbot where:

  • The user types a message in a React browser app
  • The message is sent to a Python FastAPI server
  • FastAPI calls the Groq AI API (free, fast LLM)
  • The AI response streams back word-by-word — just like ChatGPT
  • React shows each word as it arrives in real time

Why this separation of layers?

Separation of Concerns
Each file does exactly one job. api/chat.py handles HTTP — it doesn't know about Groq. ai_service.py handles Groq — it doesn't know about HTTP. This means if you switch from Groq to OpenAI, you only change ai_service.py. Nothing else breaks.

What is CORS and why do we need it?

Your browser runs React on port 5173. FastAPI runs on port 8000. Browsers block requests between different ports by default (a security feature called the Same-Origin Policy). CORS middleware in FastAPI tells the browser: "it's okay to accept requests from port 5173."

⚠️ Common Beginner Mistake
If you forget to add CORS middleware, your React app will get a "CORS error" in the browser console and the API call will fail silently. The backend is actually running fine — the browser just refuses to pass the response to JavaScript.

What is the Vite proxy?

In vite.config.js we have:

vite.config.js
proxy: {
  '/api': {
    target: 'http://localhost:8000',
    changeOrigin: true,
  }
}

This means: when React calls /api/chat/stream, Vite quietly redirects it to http://localhost:8000/api/chat/stream. The React code never has a hardcoded port number. In production, you'd use a real domain name instead.

How a Message Flows

Step by step — what happens from the moment you press Enter to seeing the AI response.

1
User presses Enter in ChatWindow.jsx
The handleSend() function runs. It reads the textarea value, then calls sendMessage(text) from the useChat hook. The textarea is cleared immediately so the user sees feedback right away.
handleSend() → sendMessage(text)
2
useChat hook adds user message to Zustand store
addMessage('user', text) is called. Zustand updates its state object. React automatically re-renders ChatWindow. The user's message bubble appears instantly — before any AI response arrives.
store.messages = [...messages, { role: 'user', content: text }]
3
An empty assistant bubble is created
startAssistantMessage() adds an empty { role: 'assistant', content: '' } to the store. isLoading is set to true. React renders an empty assistant bubble — this is the bubble that will get filled in as tokens arrive.
store.messages = [..., { role: 'assistant', content: '' }]
4
fetch() POST request sent to FastAPI
streamChat() in api/chat.js calls fetch('/api/chat/stream') with the full conversation history as JSON. This is a regular HTTP POST — but we're going to read the response body as a stream instead of waiting for it to finish.
fetch('/api/chat/stream', { method: 'POST', body: JSON.stringify({ messages }) })
5
FastAPI validates the request with Pydantic
FastAPI automatically checks that the request body matches the ChatRequest schema (has a messages array, each with role and content strings). If anything is wrong, it returns a 422 error automatically. No manual validation code needed.
ChatRequest.messages: List[Message] ← auto-validated
6
ai_service.py calls Groq with streaming enabled
stream_chat_response() calls client.chat.completions.create(..., stream=True). Groq starts generating the response. Instead of waiting for the full response, it sends tokens one at a time as they're generated.
client.chat.completions.create(model=..., stream=True)
7
Each token is wrapped as an SSE event and sent
For every text chunk Groq yields, api/chat.py wraps it as data: hello\n\n and sends it immediately via StreamingResponse. The HTTP connection stays open — it's like a one-way pipe from server to browser.
yield f"data: {chunk}\n\n" ← SSE format
8
React reads each SSE chunk from the stream
api/chat.js uses response.body.getReader() to read the stream chunk by chunk. For each chunk it receives, it strips the data: prefix and calls onChunk(text).
const { done, value } = await reader.read()
9
Zustand appends the chunk to the assistant bubble
appendToLastMessage(chunk) finds the last message in the array (the empty assistant one) and appends the new text to its content. React sees the state changed and re-renders just that bubble — the new word appears.
messages[last].content += chunk → React re-renders
10
Groq sends [DONE], streaming ends
When Groq finishes generating, our backend sends data: [DONE]\n\n. React receives it, calls onDone(), which calls finishStreaming() in the store — setting isLoading = false. The input box re-enables.
data: [DONE] → isLoading = false → input re-enabled

main.py

The FastAPI app entry point. This is where the server is created and configured.

Python Entry Point
backend/app/main.py
from fastapi import FastAPI
Import the FastAPI class. FastAPI is the framework — like a toolbox. We import just what we need. Alternative: Flask uses from flask import Flask — very similar pattern.
from fastapi.middleware.cors import CORSMiddleware
Import CORS middleware. CORS = Cross-Origin Resource Sharing. Without this, the browser blocks our React app from talking to FastAPI because they run on different ports. Middleware is code that runs on every request before it reaches our route handlers.
from app.api import chat
Import our chat routes file. This works because we have __init__.py files that make the folders into Python packages. app.api.chat means: inside the app package, inside the api package, the chat.py module.
app = FastAPI(title="GenAI Chatbot API", version="1.0.0")
Create the FastAPI application instance. This is the main app object. title and version appear in the auto-generated docs at /docs. This is similar to how you'd do app = Flask(__name__) in Flask.
app.add_middleware( CORSMiddleware, allow_origins=["http://localhost:5173"], allow_credentials=True, allow_methods=["*"], allow_headers=["*"], )
Add CORS middleware to allow React to call the API. allow_origins lists which URLs are permitted — in production, change this to your real domain (e.g. https://myapp.com). "*" for methods/headers means allow everything. Warning: Never use allow_origins=["*"] in production with allow_credentials=True — it's a security risk.
app.include_router(chat.router, prefix="/api", tags=["chat"])
Register the chat routes with a /api prefix. chat.router is an APIRouter object defined in api/chat.py. The prefix="/api" means all routes in chat.py automatically get /api prepended — so /chat/stream becomes /api/chat/stream. tags just organizes the auto-docs.
@app.get("/") async def root(): return {"message": "API is running ✅"}
A simple health check route. The @app.get("/") decorator means: "when someone makes a GET request to /, run this function". Returns a JSON object automatically — FastAPI converts Python dicts to JSON. Useful to test if the server started correctly.

config.py

Reads environment variables from the .env file and exposes them as a typed settings object.

Python Configuration
⚠️ Why not just hardcode the API key?
If you hardcode api_key = "gsk-abc123..." directly in your code and push to GitHub, your key is public and anyone can use it (racking up charges). Environment variables keep secrets out of your code.
backend/app/core/config.py
from pydantic_settings import BaseSettings
Import BaseSettings from pydantic-settings. This is a special Pydantic class that reads from environment variables. Note: in older Pydantic v1, this was from pydantic import BaseSettings. In Pydantic v2 (what we use), it moved to its own package pydantic-settings. Alternative: use os.environ.get('API_KEY') — works but no type checking.
class Settings(BaseSettings): api_key: str = "your-key-here" ai_model: str = "llama-3.1-8b-instant" max_tokens: int = 1024
Define all settings with types and defaults. Pydantic reads the .env file and matches keys by name (case-insensitive). If API_KEY=abc is in your .env, it overwrites the default. The type annotation (str, int) means Pydantic will automatically convert and validate the type — if you put a word where an int is expected, it raises an error.
class Config: env_file = ".env"
Tell Pydantic where to find the .env file. This inner Config class is a Pydantic convention. Modern alternative: In Pydantic v2 you can use model_config = SettingsConfigDict(env_file='.env') — but the nested Config class still works.
settings = Settings()
Create one shared settings instance. This runs once when Python imports the file. Every other file imports this settings object — they all share the same instance. This is called the Singleton pattern. Alternative: pass settings as function arguments — but that's tedious across many files.

schemas/chat.py

Defines the shape (structure) of data going into and out of the API using Pydantic models.

Python Data Validation
💡 What is a Schema?
A schema is a blueprint for data. It says "a Message must have a role field (string) and a content field (string)." If the frontend sends data in the wrong shape, Pydantic automatically rejects it with a clear error message — before your code even runs.
backend/app/schemas/chat.py
from pydantic import BaseModel from typing import List
Import BaseModel and List. All Pydantic data models inherit from BaseModel. List from typing is how you say "a list of X" in Python. Modern note: In Python 3.9+, you can write list[Message] directly instead of List[Message]. Either works in Python 3.9+.
class Message(BaseModel): role: str # "user" or "assistant" content: str # the message text
A single chat message. Both role and content are required strings. If either is missing or not a string, Pydantic raises a validation error. The comments explain valid values — but Pydantic doesn't enforce comment rules. In Phase 3, you might add from typing import Literal and type it as role: Literal['user', 'assistant'] to enforce it.
class ChatRequest(BaseModel): messages: List[Message] system_prompt: str = "You are a helpful AI assistant."
The request body shape for POST /api/chat/stream. messages is required (no default). system_prompt has a default value — so the frontend doesn't have to send it every time. FastAPI automatically reads this from the HTTP request body and validates it. You never write request.json() manually.

What the JSON looks like

This is what React sends in the HTTP request body:

HTTP Request Body (JSON)
{
  "messages": [
    { "role": "user", "content": "Hello!" },
    { "role": "assistant", "content": "Hi! How can I help?" },
    { "role": "user", "content": "What is Python?" }
  ],
  "system_prompt": "You are a helpful AI assistant."
}

Notice the full conversation history is sent every time — not just the latest message. This is because LLMs are stateless: they have no memory between calls. Sending all previous messages gives the AI context.

ai_service.py ★

The heart of the backend. This is where Python talks to Groq's AI model and gets a streaming response.

Python Groq Streaming
backend/app/services/ai_service.py
from groq import Groq
Import the Groq SDK client class. The groq Python package was installed via pip install groq. This one class gives us everything we need to talk to Groq's API. If you were using Anthropic: from anthropic import Anthropic. If OpenAI: from openai import OpenAI. All three follow the same pattern.
from typing import AsyncGenerator, List
Import type hints for the function signature. AsyncGenerator[str, None] means "an async function that yields strings." This is just documentation — Python doesn't enforce it at runtime, but it helps you and your editor understand the code. In Python 3.9+ you could import from collections.abc instead.
client = Groq(api_key=settings.api_key)
Create the Groq client once, at module load time. This client object holds the connection configuration. Creating it once (not inside each function) is more efficient — you're not re-reading the API key every request. This is called a module-level singleton.
async def stream_chat_response( messages: List[Message], system_prompt: str = "...", ) -> AsyncGenerator[str, None]:
Define an async generator function. async means this function can be paused while waiting for I/O (like a network call). The return type AsyncGenerator[str, None] means it yields strings. A function that uses yield is called a generator — it produces values one at a time instead of returning them all at once.
groq_messages = [ {"role": "system", "content": system_prompt}, *[{"role": msg.role, "content": msg.content} for msg in messages] ]
Convert our Message objects to Groq's expected format. Groq (and most LLM APIs) want the system prompt as a special first message with role: "system". The *[...] syntax spreads the list comprehension into the outer list. A list comprehension ([... for x in y]) is a compact way to transform a list — equivalent to a for loop that builds a new list.
stream = client.chat.completions.create( model=settings.ai_model, messages=groq_messages, max_tokens=settings.max_tokens, stream=True, )
Call Groq's API with streaming enabled. stream=True is the key — without it, this would wait for the full response before returning anything. With it, Groq starts sending tokens immediately as they're generated. The stream variable is an iterable that yields chunks.
for chunk in stream: text = chunk.choices[0].delta.content if text: yield text
The streaming loop — the most important part. for chunk in stream iterates over each piece as it arrives from Groq. chunk.choices[0].delta.content is the path to the text inside Groq's response object (choices is an array because you could request multiple responses; delta is the new portion of text; content is the string). The final chunk often has content = None, so we guard with if text. yield is what makes this a generator — it pauses the function and hands the value to the caller, then resumes from this exact line next time.
🔑 Key Concept: yield vs return
return exits the function and gives one value back.

yield pauses the function, gives one value back, then resumes from the same spot when the caller asks for the next value. This is how Python generators work — perfect for streaming because you process one chunk at a time without loading everything into memory.

api/chat.py ★

The FastAPI route handlers. Receives HTTP requests, calls the AI service, and streams back SSE events.

Python SSE HTTP Routes
backend/app/api/chat.py
router = APIRouter()
Create an APIRouter instance. A router is like a mini app — it groups related routes together. We import this router into main.py and register it with a prefix. This keeps main.py clean and makes routes easy to find. Alternative: define routes directly on app in main.py — works but gets messy as the app grows.
@router.post("/chat/stream") async def chat_stream(request: ChatRequest):
Define the streaming endpoint. @router.post means this handles POST requests. The full URL is /api/chat/stream (because main.py adds /api prefix). FastAPI sees the request: ChatRequest parameter and automatically reads + validates the JSON body. No manual request.json() needed.
async def generate(): try: async for chunk in stream_chat_response(...): escaped = chunk.replace("\n", "\\n") yield f"data: {escaped}\n\n" yield "data: [DONE]\n\n" except Exception as e: yield f"data: [ERROR] {str(e)}\n\n"
Inner generator function that wraps chunks as SSE events. SSE format requires each message to be data: content\n\n (double newline to end the event). We escape newlines in the content (\n → \\n) so they don't break the SSE format. The try/except sends errors as SSE events so React can display them. async for is used because stream_chat_response is an async generator.
return StreamingResponse( generate(), media_type="text/event-stream", headers={"Cache-Control": "no-cache"} )
Return a StreamingResponse. Unlike a normal response that sends everything at once, StreamingResponse keeps the HTTP connection open and sends data as the generator yields it. media_type="text/event-stream" is the official MIME type for SSE — this tells the browser it's receiving a stream of events. Cache-Control: no-cache prevents any proxy or browser from buffering the stream.

chatStore.js ★

The global state container for the entire conversation. Built with Zustand.

React State Management
💡 What is state?
State is data that can change over time and whose changes should update the UI. Examples: the list of messages, whether the AI is loading, any error message. When state changes, React automatically re-renders the affected components.
💡 Why Zustand instead of useState?
useState is local to one component. If ChatWindow and a Header both need to know if the AI is loading, you'd have to "lift state up" through parent components — messy with prop drilling. Zustand creates a global store any component can access directly.

Alternatives: Redux (very powerful but complex), Context API (built into React but causes performance issues with frequent updates), Jotai (similar to Zustand but atom-based).
frontend/src/store/chatStore.js
import { create } from 'zustand'
Import the create function from Zustand. create is the only thing you need from Zustand. You pass it a function that returns your state and actions.
const useChatStore = create((set, get) => ({ messages: [], isLoading: false, error: null,
Create the store with initial state. set is a function to update state — you call set({ key: value }). get reads current state without causing a re-render — useful inside action functions. The initial values ([], false, null) are what the store starts with.
addMessage: (role, content) => { set((state) => ({ messages: [...state.messages, { role, content }], })) },
An action that adds a message. We pass a function to set (instead of an object) when the new state depends on the previous state. state is the current state. [...state.messages, newMsg] uses the spread operator to create a NEW array with the new message added — we never mutate the existing array directly. Why not push? React needs new object references to detect changes. Mutating the array directly won't trigger re-renders.
appendToLastMessage: (chunk) => { set((state) => { const messages = [...state.messages] const last = messages[messages.length - 1] messages[messages.length - 1] = { ...last, content: last.content + text } return { messages } }) },
The streaming action — called on every SSE token. We copy the messages array ([...state.messages]), then copy the last message object (...last) but with updated content. This creates new references at every level — required for React to detect the change. This function runs dozens or hundreds of times per response as tokens stream in.

api/chat.js ★

Handles the streaming fetch request from the browser to FastAPI. Reads SSE chunks in real time.

JavaScript SSE Fetch API
frontend/src/api/chat.js
const response = await fetch(`/api/chat/stream`, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ messages, system_prompt }), })
Make a POST request with fetch. await pauses execution until the response headers arrive (not the body). Content-Type: application/json tells FastAPI the body is JSON. JSON.stringify converts the JavaScript object to a JSON string. Why not Axios? Axios doesn't support reading streaming responses as a ReadableStream. We use native fetch for this specific use case.
const reader = response.body.getReader() const decoder = new TextDecoder()
Get a reader for the response body stream. response.body is a ReadableStream — it arrives byte by byte. getReader() gives us a controller to read it chunk by chunk. TextDecoder converts raw bytes (Uint8Array) into a JavaScript string — because HTTP data is bytes, not text. Alternative: Use the EventSource browser API — but it only supports GET requests. Our POST with JSON body requires fetch instead.
while (true) { const { done, value } = await reader.read() if (done) break const text = decoder.decode(value, { stream: true })
Read chunks in an infinite loop until done. reader.read() returns a Promise that resolves with { done, value }. When the server closes the connection, done becomes true and we break. { stream: true } in decode tells the TextDecoder that more bytes may come — important for multi-byte characters (emoji, accented letters) that might arrive split across chunks.
const lines = text.split('\n\n') for (const line of lines) { if (!line.startsWith('data: ')) continue const data = line.replace('data: ', '').trim()
Parse the SSE format. SSE events are separated by double newlines. We split on \n\n to get individual events, then check each starts with data: . The continue keyword skips to the next loop iteration. We strip the data: prefix to get the actual content. Multiple events can arrive in one network chunk — that's why we loop through lines.
if (data === '[DONE]') { onDone(); return } if (data.startsWith('[ERROR]')) { onError(...); return } if (data) { onChunk(data) }
Handle the three possible SSE messages. [DONE] = streaming complete, call onDone(). [ERROR] = something went wrong, call onError(). Any other non-empty data = a text token, call onChunk(). These callback functions (onChunk, onDone, onError) are passed in from the useChat hook — this keeps api/chat.js decoupled from Zustand.

useChat.js

A custom React hook that connects the Zustand store and the API layer into one clean interface for components.

React Custom Hook
💡 What is a Custom Hook?
A custom hook is a JavaScript function that starts with use and can call other hooks. It lets you extract complex logic out of components into reusable functions. Components stay simple — they just call useChat() and get everything they need.
frontend/src/hooks/useChat.js
const sendMessage = useCallback(async (userText) => { if (!userText.trim() || isLoading) return
useCallback memoizes the function so it's not recreated on every render. Without useCallback, sendMessage would be a new function reference every render, causing infinite re-renders if passed to child components. The guard !userText.trim() || isLoading prevents sending empty messages or sending while already loading.
addMessage('user', userText) startAssistantMessage() const currentMessages = getMessages()
Why call getMessages() after updating state? React state updates are asynchronous — messages from useChatStore won't include the new user message yet in this same synchronous block. getMessages() calls Zustand's get() internally, which reads the store synchronously after the update. This ensures we send the complete history including the just-added message.
await streamChat({ messages: currentMessages, onChunk: appendToLastMessage, onDone: finishStreaming, onError: setError, })
Pass store actions as callbacks to streamChat. streamChat doesn't know about Zustand — it just calls whatever functions you give it. This is dependency injection: the API function is independent, and the hook wires it up to the store. If you wanted to log every chunk to the console instead, you'd just change onChunk.

ChatWindow.jsx

The main UI component. Renders the message list, handles input, and auto-scrolls during streaming.

React Component
frontend/src/components/ChatWindow.jsx
import { useState, useRef, useEffect } from 'react'
Three React hooks used here. useState — manages the input text (local to this component). useRef — gets a reference to a DOM element (the invisible div at the bottom) for scrolling. useEffect — runs code after render (to trigger auto-scroll when messages change).
const messagesEndRef = useRef(null)
A ref points to a real DOM element. We put this on an invisible div at the bottom of the message list. When we call messagesEndRef.current?.scrollIntoView(), the browser scrolls to that element. Unlike state, changing a ref doesn't cause a re-render.
useEffect(() => { messagesEndRef.current?.scrollIntoView({ behavior: 'smooth' }) }, [messages])
Auto-scroll on every message change. useEffect runs after every render where messages changed (the dependency array [messages]). During streaming, messages changes dozens of times per second (each token) — so this scrolls continuously as the AI types. ?.scrollIntoView is optional chaining — safe if the ref isn't mounted yet.
const handleKeyDown = (e) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault() handleSend() } }
Enter sends, Shift+Enter adds a new line. e.preventDefault() stops the default browser behavior (which would add a newline). We check !e.shiftKey so that Shift+Enter still works for multi-line input. This is the standard chat input behavior users expect.

How React renders

React builds a virtual DOM (a JavaScript object tree representing the UI). When state changes, React compares the new virtual DOM to the old one, finds the differences, and updates only those specific DOM elements. This is called reconciliation — it's why React is fast even with frequent updates like streaming tokens.

Key Concepts

The fundamental ideas you need to understand the whole project.

Async / Await

What it means
JavaScript and Python are single-threaded — they can only do one thing at a time. async/await allows code to pause while waiting for slow operations (network calls, file reads) without blocking everything else. await pauses that function only — other code can run in the meantime.
Sync vs Async comparison
// SYNC: blocks everything for 2 seconds
const data = slowNetworkCall()  // freezes the browser tab!

// ASYNC: pauses only this function
const data = await slowNetworkCall()  // other code keeps running

Generators (Python yield)

What it means
A generator function uses yield instead of return. Each time you call next() on it, it runs until the next yield, gives you that value, then pauses. Perfect for streaming — process one token at a time without loading the whole response into memory.

Tokens (LLM concept)

Large language models don't process whole words — they process tokens. A token is roughly 3-4 characters. "Hello world" is 2 tokens. "antidisestablishmentarianism" might be 6 tokens. Models are priced per token. max_tokens=1024 limits how long the response can be.

Immutability in React

⚠️ Critical React Rule
Never mutate state directly. Always create new objects/arrays. 
// ❌ WRONG — React won't detect this change
state.messages.push(newMessage)

// ✅ CORRECT — creates a new array
[...state.messages, newMessage]
React uses reference equality to detect changes. If you mutate an existing array, the reference is the same object — React thinks nothing changed.

The Python __init__.py file

An empty file called __init__.py inside a folder tells Python: "treat this folder as a package." Without it, from app.api import chat would fail with a ModuleNotFoundError. In modern Python (3.3+) with "namespace packages," you can sometimes skip it — but explicit __init__.py files are still the clearest approach.

Environment Variables & .env

A .env file stores secrets (API keys, database passwords) outside your code. The python-dotenv package reads this file and loads the values as environment variables. Your code reads them via os.environ or Pydantic's BaseSettings. Never commit .env to Git — add it to .gitignore.

How JWT auth works (preview)

1
User logs in with email + password
React sends POST to /api/auth/login. FastAPI checks the password hash against the database.
2
FastAPI returns a JWT token
A JWT (JSON Web Token) is a signed string containing the user's ID and role. It's signed with a secret key — can't be faked.
3
React stores the token in Zustand (authStore)
Every subsequent API request includes the token in the Authorization: Bearer token header via an Axios interceptor.
4
FastAPI validates the token on every protected route
A get_current_user dependency reads the token, validates the signature, extracts the role, and rejects the request if anything is wrong.
